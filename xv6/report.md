## Part A: `getreadcount` System Call Implementation

To track the total number of bytes read across the entire system, a new system call, `getreadcount`, was implemented.

### Kernel Modifications

1.  **Global Counter and Lock:** A global unsigned integer, `total_read_count`, was added in `proc.c` to act as the system-wide counter. To prevent race conditions on multi-core systems, a `struct spinlock read_count_lock` was also added. Using `uint` naturally handles the overflow requirement by wrapping around to zero. An initialization function, `readcountinit()`, was created and called from `main()` to set up the lock.

2.  **Modifying `sys_read()`:** The core `sys_read()` function in `sysfile.c` was modified. After a successful `fileread` operation, it now acquires `read_count_lock`, adds the number of bytes read (`n`) to `total_read_count`, and then releases the lock.

3.  **New System Call `sys_getreadcount()`:** This new function was implemented in `sysproc.c`. It safely reads the value of the global counter by acquiring and releasing the spinlock and returns the value to the user program.

4.  **System Call Hookup:** The new system call was assigned number 22 in `syscall.h` and registered in the `syscalls` array in `syscall.c`. The necessary user-space stubs were generated by updating `usys.pl` and `user.h`.

## Part B: Scheduler Implementation

The xv6 `Makefile` was modified to accept a `SCHEDULER` variable (`FCFS` or `CFS`). This adds a corresponding preprocessor flag (`-DFCFS` or `-DCFS`) during compilation, allowing for conditional compilation of the scheduler logic in `proc.c`.

### First Come, First Serve (FCFS) Scheduler

1.  **`struct proc` Modification:** A new field, `uint creation_time`, was added to `struct proc` in `kernel/proc.h` to store the process's creation timestamp.

2.  **Initialization:** In `allocproc()`, this `creation_time` is initialized with the current value of the global `ticks` variable, effectively timestamping each new process.

3.  **Scheduler Logic:** Inside the `scheduler()` function, the FCFS logic (within an `#if defined(FCFS)` block) iterates through the process table to find the `RUNNABLE` process with the smallest `creation_time`. This "oldest" runnable process is then selected for execution. It runs until it yields, sleeps, or terminates.

### Completely Fair Scheduler (CFS)

1.  **`struct proc` Modification:** Two new fields were added to `struct proc`: `int nice` (for priority, defaulting to 0) and `uint vruntime` (virtual runtime).

2.  **Virtual Runtime Tracking:**
    * **Initialization:** In `allocproc()`, `vruntime` is initialized to 0 for every new process.
    * **Updating:** A small modification was made to the timer interrupt handler in `trap.c`. On every timer tick, if a process is in the `RUNNING` state, its `vruntime` is incremented by 1. This serves as a simplified measure of its CPU consumption.

3.  **Scheduler Logic:** The CFS implementation (within an `#elif defined(CFS)` block) always chooses the `RUNNABLE` process with the **minimum `vruntime`**. It iterates through the process table, finds this process, and schedules it. This simple rule ensures that processes that have run less (have a lower `vruntime`) are prioritized, leading to a fair distribution of CPU time.

### Performance Comparison

A test program, `schedulertest`, which forks several CPU-intensive child processes, was used to compare the schedulers.

* **Round Robin (RR):** Provided a baseline with relatively fair distribution of CPU time. Waiting times were moderate for all processes as no process was allowed to starve, but frequent context switches introduced overhead.

* **First Come, First Serve (FCFS):** Performed poorly in mixed scenarios. If a long-running process started first, any subsequent short jobs experienced extremely high waiting times (the "convoy effect"). This resulted in a very high average waiting time.

* **Completely Fair Scheduler (CFS):** Demonstrated the best performance in terms of fairness and system responsiveness. By always prioritizing the process that had run the least (`vruntime`), it ensured that short jobs finished quickly without being stuck behind long ones, leading to lower average waiting and turnaround times compared to FCFS. It provides a more balanced and efficient CPU allocation than the simple Round Robin scheduler.
